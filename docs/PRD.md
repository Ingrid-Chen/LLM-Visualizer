## 1. 项目背景与目标

### 1.1 项目缘起

LLM 已经成为大众工具，但绝大多数普通用户和初阶产品经理对"LLM 内部到底在做什么"的理解仍停留在黑盒层面。市面上的科普资源呈现两极分化：

- **专业向**：Andrej Karpathy 系列视频、Jay Alammar 的图解 Transformer、Sebastian Raschka 的《Build a Large Language Model (From Scratch)》——质量极高，但对完全没有技术背景的读者门槛偏高
- **大众向**：各种"3 分钟看懂 ChatGPT"短视频——降低了理解门槛，但准确性参差，且几乎不涉及 PM 在实际工作中需要关心的参数和决策

本项目希望填补**中间地带**：用小白能进入的方式，把 LLM 工作机制讲到 PM 工作所需的深度，并在每个机制点上明确指出"作为产品经理，看到这个机制应该想到什么"。

### 1.2 项目目标与优先级

项目同时服务三个目标，按优先级排序如下：

**P0 · 求职展示（最高优先级）**

向 AI 应用方向的招聘方证明：作者具备从模型机制到产品决策的完整链路理解能力，不只是"用过 ChatGPT 的产品经理"。具体目标读者是新加坡及国内 AI 应用公司、AI Infra/开发者工具公司、传统大厂 AI 业务线的招聘负责人。

**P1 · 帮朋友入门**

为身边对 AI 感兴趣但毫无技术背景的朋友提供一个友好、有趣、严谨的入门入口。让他们看完之后既理解了基本概念，也具备了进一步学习的兴趣和方向。

**P2 · 自我汇总**

将作者在 LLM 学习过程中形成的理解结构化沉淀下来。这一目标主要通过项目副产品（PRD、内容大纲、代码注释）实现，而不是网站本身的显式产出。

### 1.3 优先级冲突的处理原则

三个目标之间存在张力，最典型的是 **"绝对严谨"与"小白友好"** 的冲突。处理原则：

- 当严谨与易懂冲突时，**保留严谨，调整表达方式**（用比喻、图示、交互替代直接展示公式），但**永远不允许为了易懂而牺牲事实正确性**
- 所有为了便于理解而做的简化或近似，**必须明确标注**"这是一个简化的说法，更精确的解释见 [链接]"
- 当某些内容确实无法在小白层做到既严谨又易懂时，宁可在小白层放一句"这部分需要一些数学背景，建议先跳过，或在深入模式中查看"，也不写一个不严谨的版本

## 2. 求职目标的具体化

将 P0 目标进一步具体化，因为它直接影响内容深度的决策。

### 2.1 目标岗位类型

按作者的兴趣排序：

1. **AI 应用 PM**（最优先）——做 LLM 落地产品的公司，比如 AI 写作、AI 客服、AI Agent 工具
2. **AI Infra / 开发者工具 PM**——做 API、平台、开发者产品的公司
3. **传统大厂 AI 业务线 PM**——在已有产品里加 AI 的业务线

模型层 PM（基础模型公司的 PM）暂不作为目标。

### 2.2 招聘方阅读场景假设

假设招聘方第一次访问网站的场景是：在简历筛选阶段，点击作品集链接，**最多停留 3-5 分钟**。在这段时间里，他需要快速判断"这个 PM 对 LLM 的理解到了什么深度"。

这意味着首页和最先看到的内容必须**信息密度足够高**，能在 3-5 分钟内传递三个信号：

- **机制理解信号**——能看出作者真懂 LLM 内部在做什么，不是表面套话
- **PM 视角信号**——能看出作者把每个机制点都翻译成了产品决策语言
- **产品 sense 信号**——网站本身的设计和叙事就是一个 PM 作品

### 2.3 与同类内容的差异化

市面上 LLM 科普资源已经非常多。本项目的差异化定位是：**不是又一个"讲清楚 LLM 怎么工作"的科普网站，而是"讲清楚 LLM 怎么工作，以及 PM 应该如何思考它"的产品**。

具体差异点：
- 每个机制模块都有 **"PM 视角"卡片**，明确把机制点连接到产品决策（如 tokenization → API 成本核算与字符限制 bug；attention → 长 prompt 中"模型忘了前面的指令"的成因）
- 视觉风格选择**温暖科普风**而非主流的工程深色风，与典型 AI 工具站做出区隔
- 用户输入贯穿全流程的**统一交互骨架**，把 7 步抽象机制变成一个连贯的故事

## 3. 用户与阅读路径

### 3.1 三类目标用户

| 用户类型 | 特征 | 来访目的 | 期望停留时长 |
|---|---|---|---|
| 招聘方 | 已具备 LLM 知识，时间紧 | 评估作者水平 | 3-5 分钟 |
| 入门朋友 | 无技术背景，对 AI 好奇 | 看懂 LLM 是什么 | 多次访问，每次 5-10 分钟 |
| 已入门读者 | 已经看过基础科普，想再深一层 | 学习更细的机制和应用层考虑 | 不定，按需查阅 |

### 3.2 三层阅读路径

为了同时服务这三类用户，网站采用**三层阅读路径**设计，底层共享同一套交互 demo，上层差异在于呈现方式：

**导览模式**（默认首页呈现）

服务对象：招聘方与快速浏览者。

设计原则：
- 首屏即出现最具视觉冲击力的交互 demo
- 每个模块只展示**最精华的 PM 视角卡片**
- 不展示给小白的铺垫文字
- 全站可在 5 分钟内扫完

**入门模式**

服务对象：无技术背景的朋友。

设计原则：
- 线性叙事结构，按章节顺序推进
- 每段内容控制在 5 分钟独立闭环
- 大量比喻、生活化例子、可玩的交互
- 每段结尾有"看懂了吗"的简单自测或复述提示，强化记忆

**深入模式**

服务对象：已入门、想再学一层的读者。

设计原则：
- 不是单独的模式，而是嵌入在入门模式中的"想了解更多"展开层
- 包含更技术的解释、关键数学概念的指引、原始论文与视频链接
- 鼓励用户跳出本网站去看 Karpathy、Jay Alammar、Sebastian Raschka 等更深层资源

**三层共享同一套交互 demo**——demo 本身不变，变的是它周围的解释文字、文案密度和上下文铺垫。这是工程上保持轻量、内容上保持一致的关键设计。

### 3.3 渐进披露原则

整站遵循 progressive disclosure 设计原则：

- 默认露出最核心、最简化的信息
- 通过"点击展开""hover 提示""链接跳转"等方式让用户**自主选择是否进入下一层**
- 永远不在页面初始状态强制展示复杂内容

## 4. 内容架构

### 4.1 内容主线选择

LLM 相关知识有两条主线：**模型如何工作**（推理流程）和**模型如何训练**（训练流程）。

**MVP 阶段先做"工作流程"这条线**。决策依据：

1. 从求职价值看，AI 应用 PM 日常打交道的就是推理侧参数（prompt、token、temperature 等），招聘方对这条线的反馈最直接
2. 从用户认知看，所有人都用过 ChatGPT，对"输入文字、出来文字"有体感，从已有体感切入认知成本最低
3. 从交互可玩性看，推理侧的参数调整能在毫秒级看到效果反馈，训练侧的参数反馈周期长、难做交互
4. 从作者自身把握度看，作者目前对推理流程的理解最扎实，先做有把握的部分能更好守住"事实严谨"的红线

训练流程作为第二期内容，待作者学习沉淀后再做。

### 4.2 工作流程的 8 个模块与优先级

将"输入到输出"完整流程拆解为 8 个模块，按重要程度分三档：

| 模块 | 优先级 | 核心要讲的事 | PM 视角的核心连接点 |
|---|---|---|---|
| 1. Tokenization | 重点 | 文字如何变成 token | API 成本核算、字符限制 bug、上下文窗口实际意义 |
| 2. Embedding | 轻量 | token 如何变成向量 | （以概念示意为主） |
| 3. Positional Encoding | 轻量 | 位置信息如何加进去 | （以概念示意为主） |
| 4. Transformer (Attention + FFN) | 重点 | 模型如何"理解"上下文 | 长 prompt 失效、heads/layers 影响、QKV 关系 |
| 5. 输出层 / Logits | 轻量 | 模型如何算出"下一个词的概率分布" | （为 Sampling 模块做铺垫） |
| 6. Sampling | 重点 | 如何从概率分布中挑词 | 不同产品场景的 temperature/top-k/top-p 选型 |
| 7. Detokenization + 循环 | 普通 | token 拼回文字、再喂回去预测 | max_tokens、streaming 的产品体验影响 |
| 8. Prompt 结构 | 普通-重点 | system / user / assistant 三种角色 | 角色优先级、prompt 注入、安全防护 |

**重点模块**承担网站最核心的展示价值，必须做精做深，包含完整的交互 demo 和 PM 视角卡片。**普通模块**做基础交互，PM 视角卡片可选。**轻量模块**主要做概念示意图和文字解释，不强求做交互。

### 4.3 统一交互骨架

整站采用**用户输入贯穿全流程**的设计：

- 用户在网站起始处输入一段文本（可以选预设例子，也可以自由输入）
- 这段文本作为整站的"主角"，跟随读者一起经过 8 个模块
- 在每个模块中，读者看到的都是**自己的输入文字在这一步发生了什么变化**
- 顶部固定保留原始输入与当前进度条

这个设计的价值：
- 把 8 个抽象步骤变成一个**连贯的故事**而不是零散的科普集合
- 招聘方一眼能看出网站的整体性叙事结构（这本身是产品 sense 的体现）
- 读者全程有"这是我自己的文字在被处理"的代入感

### 4.4 各模块详细设计

#### 模块 1：Tokenization（重点）

**为什么放重点：** Tokenization 是 AI 应用 PM 最常打交道的概念之一——它直接决定 API 成本、字符限制 bug 的来源、上下文窗口的实际可用空间。但绝大多数科普资源把它当作技术细节一笔带过。

**核心要传达的认知：**
- 模型不是按"字"或"单词"理解文字的，而是按 token
- 同一句话在中英文里 token 数量差异很大（典型情况是中文 token 数约为英文的 1.5-2.5 倍，具体值需通过真实测试确认而非凭印象写）
- 同一个词在不同位置可能被切成不同的 token（典型例子是前面带空格的版本）

**交互设计：**
1. 让用户输入一段中英混合 + emoji + 数字的文本
2. 实时高亮显示分词结果，每个 token 用不同颜色区分
3. 显示对比卡片：同一句话翻译成中英文后各多少 token
4. 显示成本对比：按当前主流 API 价格（需查最新数据）算出这段话的输入费用

**技术实现：** 前端用 `js-tiktoken` 库直接跑，无需 API。

**PM 视角卡片：**
- "4k 上下文窗口 ≠ 4k 中文字"——为什么这件事在产品里重要
- API 计费实例：一段中英对照客服回复的成本对比
- 字符限制 bug 的来源：当产品按字符数做截断而模型按 token 数收费时会发生什么

**深入模式延伸：** 链接到 [Tiktokenizer](https://tiktokenizer.vercel.app/) 让感兴趣的读者继续探索；附 BPE 算法简介与原始论文链接。

---

#### 模块 2 & 3：Embedding 和 Positional Encoding（轻量）

**为什么放轻量：** 这两步是机制必要的环节，但对 AI 应用 PM 的日常工作几乎不产生直接决策影响。讲清楚概念即可，不必做大型交互。

**核心要传达的认知：**
- Embedding：每个 token 被映射成一个高维向量，向量的相对位置反映了语义关系
- Positional Encoding：因为 Transformer 本身不区分词的顺序，所以需要把"位置信息"也加进去

**交互设计：** 一个简化的 2D 散点图（实际是高维降到 2D 的可视化），让用户看到几个示例词在向量空间中的相对位置。这个图可以是预生成的静态图，不必实时计算。

**注意事项：** 必须明确标注"这里展示的是降维后的简化示意，真实 embedding 维度通常是几百到几千维，无法直接可视化"——避免读者误以为模型真的在 2D 平面上比对词义。

**PM 视角卡片（可选）：** 简单提一下"语义检索 / RAG 的核心就是 embedding 的相似度计算"，作为后续模块的钩子。

---

#### 模块 4：Transformer（Attention + FFN）（重点）

**为什么放重点：** Attention 是 LLM 最难讲清楚也最值得讲清楚的部分。讲好了是网站亮点，讲不好就是又一个抄 Jay Alammar 的版本。同时这一步与许多 PM 实际遇到的问题（如长 prompt 中模型忽略前面指令）直接相关。

**关键取舍：** 不做"真实可调的 mini Transformer"——那样工程量过大，会拖垮整个项目进度。改做"**预设例子的 attention 可视化**"，用精选的几个有代表性的句子，让用户切换不同设置看到 attention 的变化。

**核心要传达的认知：**
- Attention 让每个词在被处理时"看向"句子中其他相关的词
- 多头注意力（multi-head）让模型同时从多个角度关注上下文
- 多层堆叠（layers）让模型逐层抽象出更复杂的语义关系
- QKV 三个矩阵分别承担"问什么 / 看哪里 / 取什么"的角色
- FFN 层（前馈网络）承担类似"知识库查询"的功能（注：这是主流但仍有争议的解释，文案中需说明）

**交互设计：**

1. **Attention 可视化**：展示 3-4 个精心设计的预设句子（含指代、长距离依赖、长 context 等情况），用户可以：
   - 选择"看 1 个头 vs 看多个头"
   - 选择"看第几层"
   - 鼠标悬停在某个词上，看它"看向"哪些其他词（连线粗细表示注意力权重）

2. **QKV 关系动画**：一个完全预设的小动画，演示 Q、K、V 三个矩阵如何交互。不可调节，但视觉要做到精致。

3. **FFN 模块**：用一个简化卡片说明 FFN 的作用，加一个小 demo（比如展示同一个 token 在不同 FFN 层激活了哪些"特征"）。不强求做大交互。

**PM 视角卡片：**
- 为什么模型在长 prompt 中会"忘"前面的指令——展示一个长 context 例子，让用户看到 attention 是如何分散的
- 注意：网站文案中需精确表述为"attention 是其中一个原因"而非"就是 attention 的问题"——避免事实性错误
- heads 数和 layers 数对效果的实际影响，以及在做模型选型时如何权衡

**深入模式延伸：** 链接到 Jay Alammar 的 Illustrated Transformer、Karpathy 的代码视频、bbycroft.net/llm 的 3D 可视化。

---

#### 模块 5：输出层 / Logits（轻量）

**为什么放轻量：** 这一步本身简单——模型在最后一层把内部状态映射回整个词表的概率分布。它是 Sampling 模块的输入，所以放在 Sampling 之前作为铺垫即可。

**交互设计：** 展示用户输入的最后一个位置，模型输出的 top 候选词及其原始 logit 值。可以与 Sampling 模块的演示数据共用。

---

#### 模块 6：Sampling（重点）

**为什么放重点：** Sampling 模块是整个网站最有可能产生"哇时刻"的部分——交互即时、视觉冲击强、PM 视角清晰，且与用户日常使用的 ChatGPT/Claude 的"创意 vs 严谨"的体感直接挂钩。

**核心要传达的认知：**
- 模型每一步都不是"挑一个词"，而是先算出**整个词表上的概率分布**
- 不同的采样策略（temperature / top-k / top-p）决定如何从这个分布里挑词
- temperature 拉平或锐化分布，top-k / top-p 截断分布的尾部

**关键工程取舍：** 用 OpenAI API 一次性预生成几个典型例子的真实 logprobs 数据存为 JSON 静态文件。运行时不调用 API，前端通过数学公式实时模拟不同 temperature 下的分布变化（temperature 调整在数学上等价于把 logprob 除以一个数再 softmax，这是严格成立的）。

这个方案的价值：
- 数据 100% 真实（来自真实模型的真实概率分布）
- 不需要后端、不需要付费 API、国内访问无障碍
- 是"严谨 + 工程合理"的典范，本身就是值得展示给招聘方的工程决策

**交互设计：**
1. 用户从 5-10 个预设例子中选一个（如"中国的首都是"、"今天天气真"、"我最喜欢的水果是"），不同例子刻意覆盖"高确定性"和"低确定性"两类场景
2. 展开柱状图，显示 top 20 候选词及其概率
3. 用户拖动 temperature 滑块，柱状图实时变化（前端模拟）
4. 用户切换 top-k / top-p，分布尾部对应被截断
5. 点击"采样一次"按钮，按当前分布采样出一个词，并把它接到原文后面。此后用户可继续点击"再生成一个 token"，但从第二步起进入"演示模式"——按数据生成阶段预生成的 greedy 路径展示后续逐 token 输出（每步分布仍为真实数据）。演示模式下温度滑块禁用并提示"演示模式温度固定"，避免"调温度但路径不变"的逻辑不一致。详见 `docs/Sampling-DevSpec.md` 第 6.7 节。

**PM 视角卡片：**
- 不同产品场景的 temperature 选型实例：客服机器人（低）、创意写作工具（高）、代码生成（极低）
- 为什么"高 temperature ≠ 更聪明"——它只是更随机
- 一个常被忽略的点：top-p 比 top-k 在多数应用场景下更鲁棒，原因是它能根据分布形态动态截断

**深入模式延伸：** 简介 typical sampling、min-p sampling 等较新方法；链接到相关研究论文。

---

#### 模块 7：Detokenization + 循环（普通）

**为什么放普通：** 这一步常被忽略，但它实际上是 PM 理解"流式输出"和"模型如何越说越跑偏"的关键。

**核心要传达的认知：**
- 模型一次只生成一个 token，然后把这个 token 加到输入末尾，再次预测下一个
- 这个循环一直进行，直到模型生成 EOS（结束 token）或达到 max_tokens 上限
- 因为每一步都基于之前所有输出，所以早期的小偏差会被放大（这就是"越说越跑偏"的机制根源之一）

**交互设计：**
1. 复用之前模块的预设例子，但这次让用户**逐 token 看模型生成过程**——每点击"下一个 token"按钮，输出就增加一个词
2. 提供"streaming on/off"对比演示：左边一字一字蹦出，右边等几秒后整段出现，但总耗时一样
3. 展示 max_tokens 截断效果：用户可以设一个很小的 max_tokens 值，看到模型被强行截断

**PM 视角卡片：**
- streaming 是感知性能的典型案例：总时间不变但用户感觉差很多
- streaming 的副作用：什么时候应该关掉它（如需要 JSON 解析、需要敏感词过滤后再展示）
- max_tokens 设置过小导致输出被截断的产品 bug，以及与计费的关系

---

#### 模块 8：Prompt 结构（单独成章）

**为什么单独成章：** 这一节虽然不严格属于"输入 → 输出的机制流程"，但对 AI 应用 PM 的日常工作极其重要——足够单独成为一章。把它放在最后，作为"机制讲完了，现在讲怎么用"的过渡。

**核心要传达的认知：**
- system / user / assistant 是 chat 模型中的三种角色
- 它们最终也是 token，只是前面加了特殊的角色标记
- 角色之间的"权重"不是绝对的，更多是模型在 SFT/RLHF 阶段学到的倾向
- prompt 注入攻击的根本原因就是模型很难真正区分"系统指令"和"用户输入中混入的伪装指令"

**交互设计：**
1. 一个简化的多轮对话编辑器，让用户编辑 system / user 内容，看不同设置下模型的"倾向"
2. 一个 prompt 注入演示：用户输入一段试图让模型违背 system prompt 的话，看模型如何反应（用预生成的几个真实案例对比展示）

**PM 视角卡片：**
- 角色优先级在产品安全设计中的意义
- 为什么"加一句别忘了你是客服"在 prompt 工程里有时管用有时不管用
- prompt 注入的产品级防护策略

## 5. 视觉与交互风格

### 5.1 风格定位

整体定位：**温暖科普风** + **重点模块杂志级强视觉**。

为什么这样选：

- 温暖科普风**直接对应小白友好的产品定位**——风格选择本身就是一次产品决策，能向招聘方传递"这个 PM 知道为谁设计"的信号
- 整站避免主流 AI 工具站的深色冷色工程感（如 Vercel、Linear、bbycroft.net 那种），形成**视觉记忆点**
- 但在重点模块（Tokenization、Sampling、Transformer）保留杂志级强视觉处理，确保招聘方在 3-5 分钟扫描时能感受到"主菜级别"的设计冲击

### 5.2 风格执行细节

以下为方向性建议，具体设计在实现阶段细化：

**字体策略**——避免"AI 味"的关键不在颜色而在字体。一旦用了 Inter / Geist / SF Pro 这类典型科技字体，无论什么颜色都会立刻"AI 化"。建议中文用思源宋体或空号体一类有人文感的字体作主标题；英文配合衬线字体或人文感强的无衬线。

**配色策略**——避免"温暖 = 米黄+橙红"的陷阱（容易做成母婴/烘焙/小红书种草感）。建议奶油白底 + 一个不那么饱和的主色（深绿、勃艮第红、墨蓝），点缀色再用暖橘。

**插画感不一定要画插画**——大色块、几何形状、手绘感的箭头/标注，都能营造人味，比真画插画工作量小得多。

**重点模块的杂志级处理**——不必整个换皮，保持底色和字体延续，只在那一屏加强排版张力（大标题、大留白、大图表），就会有"主菜端上来了"的感觉。

### 5.3 交互原则

- **每个交互都有反馈**——用户操作后必须有视觉变化，无论是动画、颜色变化还是数值更新
- **避免过度动画**——动画服务于理解，不为炫技而做
- **重要操作有撤销 / 重置**——比如 temperature 拉到极端值后，提供"恢复默认"
- **移动端适配** 是必须的（朋友们大概率在手机上看），但**桌面端体验是首要优化目标**（招聘方更可能在电脑上看作品集）

## 6. 工程方案

### 6.1 总体策略

**纯静态网站，无后端，无运行时 API 调用。**

这个策略的依据：
- 所有需要"真实模型数据"的部分（如 Sampling 的概率分布）都通过预生成 + 前端模拟实现
- 所有用户交互都可以在前端完成（tokenization 用 js-tiktoken，attention 可视化用预设数据，温度模拟用 JS 数学）
- 学习进度等用户数据用浏览器 localStorage 存储，不需要服务端持久化

这个策略的好处：**部署成本接近零、国内国外访问都快、无 API 兼容性风险、维护负担最小**。

### 6.2 技术栈选型

| 组件 | 选型 | 备注 |
|---|---|---|
| 前端框架 | Next.js（静态导出模式） | 与 vibecoding 工具链兼容好 |
| 样式 | Tailwind CSS | 标准选择 |
| Tokenization | js-tiktoken | 前端直接运行 |
| 数据可视化 | Recharts（备选 D3 或纯 SVG） | 先用 Recharts 跑通 v0.1，效果不满意再换 |
| 概率分布数据 | 预生成 JSON | 静态文件放 public 目录 |
| 状态管理 | React 内置 hooks 即可 | 无需 Redux 等 |
| 用户数据 | localStorage | 学习进度记录 |
| 流量统计 | Plausible 或 Umami | 轻量、隐私友好 |
| 部署 | Cloudflare Pages 或 Vercel | 国内访问情况待验证 |
| 域名 | 自购独立域名 | 提升求职作品集的专业感 |

### 6.3 数据预生成流程

Sampling 模块所需的真实概率分布通过以下流程生成：

1. 选定 5-10 个有代表性的输入文本
2. 调用 OpenAI Chat Completion API，参数设置为 `logprobs: true, top_logprobs: 20, temperature: 1`
3. 把每个例子的 top 20 候选词及其 logprob 存入 JSON 文件
4. 前端读取 JSON 文件，根据用户当前的 temperature / top-k / top-p 设置实时模拟分布变化

这个流程一次性完成，预计 API 费用控制在 1 美元以内。

### 6.4 国内国外访问方案

国内访问的现实情况复杂：
- Vercel 在国内访问稳定性有时不佳
- Cloudflare Pages 国内访问情况近年波动较大
- 部署到国内云（阿里云 / 腾讯云）需要 ICP 备案，流程较长

**初步方案**：先用 Cloudflare Pages 做主部署，观察国内访问情况；如果体验不佳，考虑接入国内 CDN 或迁移到国内云（届时再处理备案问题）。这个决定不必现在做，等真到部署阶段实测后再决定。

### 6.5 数据收集与隐私

收集的数据范围：
- **匿名访问统计**：PV / UV / 停留时长 / 来源（通过 Plausible 或 Umami）
- **学习进度**：用户看过哪些章节、是否完成（仅存浏览器 localStorage，不上传服务器）

不收集任何用户输入的文本内容，不收集任何可识别个人身份的数据。

## 7. 落地路径与里程碑

### 7.1 MVP 范围定义

MVP 范围限定为：**8 个模块中的 Sampling 模块 + 网站整体框架（首页、导航、视觉风格、统一交互骨架）**。

为什么先做 Sampling 而不是按顺序做 Tokenization：
- Sampling 模块技术风险最低（数据预生成方案已明确）
- Sampling 模块教学效果最爆（直接对应用户日常体感）
- Sampling 模块 PM 视角最清晰（每个产品场景都有明确选型逻辑）
- 用 Sampling 验证整体视觉与交互骨架后，其他模块复用难度大幅降低

### 7.2 阶段划分

**第一阶段（MVP）**

产出物：能跑通的 Sampling 模块 + 网站基础框架 + 首页。

完成标志：
- 网站可以访问
- 首页能传递三层阅读路径的结构
- Sampling 模块完整可玩
- 视觉风格基本定型

**第二阶段（核心扩展）**

依次完成：Tokenization、Transformer、Prompt 结构这三个重点模块。

**第三阶段（补完）**

完成 Embedding、Positional Encoding、输出层、Detokenization+循环这四个轻量到普通的模块，整站 8 个模块全部上线。

**第四阶段（演化）**

加入"训练流程"内容线、未来的 Memory / Agent 等扩展模块。

### 7.3 节奏建议

不设硬性 deadline。建议遵循以下节奏：
- 每个阶段开始前，先把范围内模块的具体设计文档写完
- 设计文档完成后再进入 vibecoding 实现
- 每完成一个模块就部署上线，根据真实使用反馈调整

求职用作品集时，**第一阶段完成即可作为求职材料展示**——一个有完整框架、一个精打磨的核心模块、清晰的产品规划，比一个仓促完成 8 个模块但每个都不深的网站更能体现 PM 能力。

## 8. 内容严谨性保障

考虑到"绝对不能有事实错误"是核心红线，建立以下保障机制：

### 8.1 写作流程

每个模块的内容文案按"三稿制"产出：
1. **初稿**——作者基于自己的理解写出
2. **校稿**——对照参考资料（见附录）逐句核对，特别核对每个"事实陈述句"
3. **简化稿**——在保持事实正确的前提下做小白友好化改写

### 8.2 表达分级

文案中的每一句话都属于以下三类之一，并对应不同标识：
- **精确陈述**：可直接使用，无需特殊标注
- **简化陈述**：旁边明确标注"这是一个简化的说法，更精确解释见 [链接]"
- **争议性陈述**（如 FFN 的"知识库"解释）：必须明确说明"目前主流的解释是 X，但学界仍有讨论"

### 8.3 参数与数据的来源约束

任何具体数值（如某模型的上下文窗口、某 API 的价格、某基准的得分）**必须有明确的引用来源**，并标注获取日期。如果无法找到权威来源，宁可不写。

参考作者的工作准则：**对于具体产品参数、型号规格、配置等有明确答案的问题，必须先搜索再确认，不能用经验规律猜测**。这条准则同样适用于本网站的所有内容。

## 9. 风险与权衡

### 9.1 内容严谨性 vs 易懂性

**风险：** 在追求小白友好的同时引入事实错误。

**应对：** 严格遵循第 8 节的写作流程；所有简化必须显式标注；存疑内容宁可不写。

### 9.2 工程量失控

**风险：** Transformer 模块的 attention 可视化是工程黑洞，容易吃掉过多时间。

**应对：** MVP 阶段不做 Transformer 模块，等 Sampling 验证骨架后再做；做 Transformer 时严格守住"预设例子"边界，不做实时计算。

### 9.3 与已有科普资源的差异化失效

**风险：** 做出来的版本被 Karpathy / Jay Alammar / 3Blue1Brown 等更专业的资源对比下显得平庸。

**应对：** 不与他们在"机制讲解"维度上正面竞争。本项目的差异化是**"PM 视角"和"统一交互骨架"**，这是他们没有做的角度。每个模块的 PM 视角卡片必须真实有产品价值，不能是套话。

### 9.4 招聘方实际反馈与作者预期不符

**风险：** 网站做出来后，招聘方反馈不如预期。

**应对：** 第一阶段（MVP）完成后，主动找 3-5 位 AI 应用 PM 朋友试看，收集反馈后再继续推进。在投出大量求职简历前先做小范围验证。

### 9.5 未来扩展性

**风险：** 当扩展到训练流程、Memory、Agent 等内容时，"统一交互骨架"的输入文本设计可能不再适用。

**应对：** 当前只对工作流程主线承诺统一骨架；未来扩展模块允许设计自己的交互骨架，但视觉风格保持一致。

考虑到本项目的非商业性质，不设硬性数字 KPI。但以下几类信号可作为参考：

**P0 目标（求职展示）的成功信号：**
